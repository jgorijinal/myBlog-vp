# 树结构

为什么需要使用树结构来保存数据呢?

树结构 和 数组/链表/哈希表 对比起来有什么优点呢 ?

TIP

# 数组

  1. 优点:
  • 数组的主要优点是根据 下标值访问 效率会很高
  • 但是如果希望根据元素查找对应的位置呢?
  • 比较好的方法是先对数组进行排序, 在进行 二分查找
  1. 缺点:
  • 需要先对数组进行排序, 生成有序数组, 才能提高查找效率
  • 另外数组在插入和删除数据时, 需要有大量的位移操作(比如在首位或中间位置进行添加或删除操作), 效率很低

TIP

# 链表

  1. 优点
  • 插入和删除数据的效率高
  1. 缺点:
  • 查找效率比较低, 需要重头开始依次访问链表中的每个数据项, 直到找到为止
  • 而且即使是删除或者插入的效率很高, 但是如果要插入和删除中间位置的数据, 还是需要从头先找到对应的数据

TIP

# 哈希表

  1. 优点:
  • 哈希表的插入/查询/删除 效率都是很高的
  1. 缺点
  • 空间利用率不高, 底层使用的是数组, 并且某些单元是没有被利用的
  • 哈希表中的元素是无序的, 不能按照固定的顺序来遍历哈希表中的元素
  • 不能快速的找出哈希表中的 最大值/最小猪 这些特殊的值

树结构

  1. 不能说树结构比其他数据结构都要好, 因为每种数据结构都有自己的应用场景
  2. 但是树结构确实也综合了上面数据结构的优点(当然这些优点不足于盖过其他数据结构, 比如效率在一般情况下没有哈希表高)
  3. 并且也弥补上了上面数据结构的缺点

而且为了模拟某些场景, 使用树结构会更加方便

  • 因为树结构是非线性的, 可以表示一对多的关系
  • 比如文件的目录结构

# 树的术语

树: n(n >= 0) 个节点构成的有限集合

  • 当 n = 0 时 , 称为 空树

对于任意棵非空树(n >= 0) ,它具备一下特性

  • 树中有一个称为 根 (Root) 的特殊节点, 用 r 表示
  • 其余节点可分为 m(m>0) 个互不香蕉的有限集合, T1, T2, T3 ... Tm , 其中每个集合本身又是一棵树, 称为原来树的 "子树(SubTree)"

# 二叉树的概念

如果树中的每个节点最多只能有两个节点, 这样的树就成为了 "二叉树"

二叉树的定义:

  • 二叉树可以为空, 也就是没有节点
  • 若不为空, 则它是有根节点 和称为左子树 TL 和 右子树 TR 的两个不相交的二叉树组成

# 二叉树的特性

有几个比较重要的特性, 在笔试题较常见:

  • 一颗二叉树第 i 层的最大节点为 2^(i-1)(n >= 1)
  • 深度为 k 的二叉树有最大节点总数为: 2^k - 1(n >= 1)
  • 对任何非空二叉树 T , 做 n0 表示叶节点个个数, n2 为度为 2 的非叶节点的个数 , 那么两者满足关系 n0 = n2 + 1

# 完美二叉树

完美二叉树 Perfect Binary tree ,也称为了满二叉树 Full Binary tree

  • 在二叉树中 , 除了最下一层的叶节点外 , 每层节点都有两个子节点, 就构成了满二叉树

# 完全二叉树

完全二叉树

  • 除了二叉树的最后一层外, 其他各层的节点都达到了最大个数
  • 且最后一层从左向右的叶节点连续存在, 只缺右侧若干节点
  • 完美二叉树是特殊的完全二叉树

# 二叉搜索树

二叉搜索树 (BST , Binary Search Tree) 也称 二叉排序树, 或者 二插查找树

二叉搜索树是一颗二叉树, 可以为空

如果不为空满足一下性质:

  • 非空子树的所有键值小于根节点的键值
  • 非空右子树的所有键值大于其根节点的键值
  • 左, 右子树本身也都是二叉搜索树

下面哪个属于二叉搜索树?

二叉搜索树的特点:

  • 二叉树的特点就是相对娇小的值总是会保存在左节点上, 相对较大的值总是会保存在右节点上
  • 那么利用这个特点可以做什么事情呢?
  • 查找效率非常高, 这也就是二叉搜索树中, 搜索的来源

import { btPrint } from 'hy-algokit' // 引入打印bst树的函数
class TreeNode<T> {
  value: T 
  left: TreeNode<T> | null =  null
  right: TreeNode<T> | null = null

  constructor(value: T) {
    this.value = value 
  }
}

class BSTree<T> {
  private root: TreeNode<T> | null = null 
  print() { // 控制台打印树的结构
    btPrint(this.root)
  }
  private insertNode(node: TreeNode<T>, newNode: TreeNode<T>) {  // 单独封装插入节点的步骤
    if (newNode.value < node.value) { // 去左边找空白位置
      if (node.left === null) {  // Node 节点的左边已经是空白
        node.left = newNode
      } else {
        this.insertNode(node.left , newNode)
      }
    } else { //去右边继续查找空白位置
      if (node.right === null) {
        node.right = newNode
      } else {
        this.insertNode(node.right , newNode)
      }
    }
  }


  // 二叉树相关的一些其他方法
  /** 插入数据的操作 */
  insert(value: T) {
    // 1. 根据传入的 value 创建 Node 节点
    const newNode = new TreeNode(value)
    // 2. 判断当前是否已经有根节点
    if (!this.root) { // 根节点为空
      this.root = newNode
    } else { // 树中已经为其他值
      this.insertNode(this.root ,newNode)
    }
  }
}

# 遍历二叉搜索树

树的遍历:

  • 遍历一棵树是指访问树的每个节点(也可以对每个节点进行某些操作, 当然这里就做简单的一个打印)
  • 但是树 和 线性结构不太一样, 线性结构通常按照从前到后的顺序, 但是树呢?
  • 应该是从树的顶端开始还是底端开始呢 ? 从左开始还是 从右开始呢?

二叉树常见的遍历方式有四种:

  • 先序遍历 preOrderTraverse
  • 中序遍历 inOrderTraverse
  • 后序遍历
  • 层序遍历 levelOrderTraverse

# 先序遍历

遍历过程为:

  1. 访问根节点
  2. 先序遍历其左子树
  3. 先序遍历其右子树

# 中序遍历和后序遍历

中序遍历过程:

  • 中序遍历左子树
  • 访问根节点
  • 中序遍历右子树

后序遍历过程:

  • 后序遍历其右子树
  • 后序遍历其左子树
  • 访问根节点

    this.postOrderTraverseNode(this.root)

    if (node) {
      this.postOrderTraverseNode(node.left)
      this.postOrderTraverseNode(node.right)
      console.log(node.value)
    }
  }
}

# 删除操作

二叉搜索树的删除有点复杂, 需要一点一点来完成

删除节点要从查找要删除的节点开始, 找到节点后需要考虑 三种情况:

  1. 该节点是叶节点(没有子节点, 比较简单)
  2. 该节点只有一个子节点(也比较简单)
  3. 该节点有两个子节点(情况比较复杂)

先从查找要删除的节点入手

  1. 先找到要删除的节点, 如果没有找到, 就不需要删除

    • 删除叶子结点
    • 删除只有一个子节点的节点
    • 删除有两个子节点的节点

    // 获取右子树
    let current = delNode.right
    let successor: TreeNode<T> | null = null
    
    while (current) {
      successor = current
      current = current.left
      if (current) {
        current.parent = successor
      }
    }

    // 拿到了后继节点
    if (successor !== delNode.right) {
      successor!.parent!.left = successor!.right
      successor!.right = delNode.right
    }

    // 一定要进行的操作: 将删除节点的 left , 赋值给后继节点的left 
    successor!.left = delNode.left
    return successor!
  }
  remove(value: T) {
    const current = this.searchNode(value)
    if (!current) return false

        this.root = null

        current.parent!.left = null
      } else {
        current.parent!.right = null
      }

    // 3. 只有一个节点的情况
    // (1) 只有左节点
    else if (current.right === null) {
      if (current === this.root) {
        this.root = current.left
      } else if (current.isLeft) {
        current.parent!.left = current.left
      } else {
        current.parent!.right = current.left
      }

    else if (current.left === null) {
      if (current === this.root) {
        this.root = current.right
      } else if (current.isLeft) {
        current.parent!.left = current.right
      } else {
        current.parent!.right = current.right
      }

      //4. 有两个节点
    else {
      const successor = this.getSuccessor(current)
      if (current === this.root) {
        this.root = successor
      } else if (current.isLeft) {
        current.parent!.left = successor
      } else {
        current.parent!.right = successor
      }
    }
    return true
  }

# 二叉搜索树的缺陷

二叉搜索树作为数据存储的结构有重要的优势

  • 可以快速的找到给定关键字的数据项, 并且可以快速的插入和删除数据项

但是, 二叉搜索树有一个很麻烦的问题

  • 如果插入的数据是一个有序的数据, 比如下面的情况
  • 有一颗初始的数据为 9 8 12 的二叉树
  • 插入了下面的数据 7 6 5 4 3

非平衡树:

  • 比较好的二叉搜索树数据应该是左右分布均匀的
  • 但是插入连续数据后 , 分布的不均匀, 称这种树为 非平衡树
  • 对于一颗平衡二叉树来说 , 插入/查找等操作的效率是 O(logN)
  • 对于一颗非平衡二叉树, 相当于编写了一个链表, 查找效率变成了 O(N)

# 树的平衡性

为了能以较快的时间O(logN)来操作一颗树, 需要保证树是平衡的

  • 至少大部分是平衡的, 那么时间复杂度也是接近于 O(logN)
  • 也就是说树中的每个节点的左孙子节点的个数, 应该尽可能的等于右边的子孙子节点的个数

常见的平衡树有哪些?

AVL树:

  • AVL树是最早的一种平衡树, 他有些办法保持 树的平衡(每个节点多存储了一个额外的数据)
  • 因为 AVl树是平衡的, 所以事件复杂度也是 O(logN)
  • 但是, 每次插入/删除操作对于红黑树效率都不高, 所以整体效率不如红黑树

红黑树:

  • 红黑是是通过一些特性来保持树的平衡
  • 因为是平衡树, 所以事件复杂度也是在 O(logN)
  • 另外插入/删除等操作, 红黑树的性能要由于 AVL树 , 所以现在平衡树的应用基本都是红黑树
